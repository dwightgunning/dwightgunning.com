Joe Gregorio has published the first article, How to create a REST protocol in his new column, The Restful Web. It's a great guide to help anybody get into the right frame of mind for developing RESTful services.

The article points to Paul Prescod'sÂ Common REST mistakes which I found to be very useful in the early stages of my first REST service. Point 4, do not put actions in URIs, such as action=delete, is worth noting:

First, you are using GET to do something unsafe. Second, there is no formal relationship between this "action URI" and the "object" URI. After all your "action=" convention is something specific to your application. REST is about driving as many "application conventions" out of the protocol as possible.

In contrast, Implementing REST Web Services: Best Practices and Guidelines recommends the use of query parameters, although it cautions that their use may not be entirely RESTful.

A resource may have different views, even if there is only one representation available. For example, a resource has an XML representation but different clients may only see different portion of the same XML. Another common example is that a client might want to obtain metadata of the current representation.To obtain a different view, a client can set a "view" parameter in the URI query string. For example: GET http://www.example.com/abc?view=meta

This is legitimate with respect to the URI specification as the query suffix is not actually a part of the URI... so http://mydomain.com/app/resA?view=min and http://mydomain.com/app/resA?view=max both represent the same resource.

I would probably side with prescod's advice that leaving application conventions, such as this one, out of your RESTful service will increase its usability. Interestingly there wasn't much debate about Hao He's Best Practices article amongst the REST community.
